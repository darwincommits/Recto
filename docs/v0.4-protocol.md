# Recto v0.4 — Phone-Resident Vault Protocol

**Status:** Draft for review (2026-04-26).
**Implements:** Recto v0.4 — "Secrets that never sit on the server"
(see ROADMAP.md).

This document is the wire-protocol contract between the **phone app**
(MAUI Blazor, runs on the operator's phone, holds Ed25519 private keys
in the platform's hardware enclave) and the **bootloader** (Python,
runs alongside Recto's launcher on the server, brokers sign requests
between supervised child processes and the phone). Both sides MUST
implement this contract verbatim; any deviation is a bug in one side
or the other and SHOULD surface as a protocol-version mismatch error,
not a silent failure.

## Goals

1. **Private keys never leave the phone.** Every cryptographic operation
   that uses the key happens inside the phone's Secure Enclave (iOS) or
   StrongBox (Android). The key material is non-exportable from
   software, even by the operating system.
2. **Biometric-gated sign approval.** Each sign request requires the
   operator's Face ID / Touch ID / fingerprint approval before the
   enclave releases the signature. The phone enforces this via the
   platform's `LAContext` (iOS) / `BiometricPrompt` (Android) APIs.
3. **No plaintext secret material on the server.** The server-side
   bootloader holds only the phone's public key, ephemeral session
   tokens (themselves signed by the phone), and pending sign requests.
4. **Survives offline phone.** Operations that have a valid cached
   session token proceed without phone interaction; only session
   issuance and renewal require the phone to be online and approve.
5. **Recto's existing service.yaml shape is unchanged.** A consumer
   declares `source: enclave` for any secret they want phone-gated; the
   launcher's existing SigningCapability code path (already in v0.1
   `recto.secrets.base.SigningCapability`) handles the dispatch.
   No new top-level YAML fields.

## Non-goals (v0.4.0)

- Multi-user / per-employee onboarding flow. v0.4.0 is single-operator;
  the phone app holds one key pair per service. v0.5+ adds account
  abstractions.
- App Store / Play Store distribution. v0.4.0 ships via TestFlight (iOS)
  and APK sideload (Android). The operator installs the app on their
  own phone(s) only.
- Co-signers / m-of-n thresholds. v0.4.0 is single-signer per secret.
- Cold-storage backup of phone keys. The operator is expected to hold a
  second phone with the same key material as the recovery path; mutual
  registration ceremony is documented as a runbook, not automated.
- Post-quantum signatures. The wire format leaves room for an
  `algorithm` field per signature so v0.5+ can add Dilithium / Falcon /
  SPHINCS+ once iOS Secure Enclave / Android StrongBox support catches
  up. v0.4.0 is Ed25519-only.

## Components

```
┌──────────────────────┐  HTTPS  ┌──────────────────────┐
│   Phone App          │◄────────┤  Bootloader          │
│  (MAUI Blazor)       │  push   │  (Python / Recto)    │
│                      │────────►│                      │
│  ┌────────────────┐  │         │  ┌────────────────┐  │
│  │ Secure Enclave │  │         │  │ Pending Reqs   │  │
│  │ / StrongBox    │  │         │  │ + Sessions     │  │
│  └────────────────┘  │         │  └────────────────┘  │
└──────────────────────┘         └──────────┬───────────┘
                                            │ Unix socket
                                            │ / Named pipe
                                            ▼
                                 ┌──────────────────────┐
                                 │  Launcher            │
                                 │  (Recto)             │
                                 └──────────┬───────────┘
                                            │ Unix socket
                                            │ / Named pipe
                                            ▼
                                 ┌──────────────────────┐
                                 │  Supervised Child    │
                                 │  (consumer process)  │
                                 └──────────────────────┘
```

**Phone app**: holds the Ed25519 private key inside the platform's
hardware enclave. Renders a UI for incoming sign requests with the
context (which service, which secret, what's being signed). Approves
or denies after biometric prompt. Signs and returns the signature
to the bootloader.

**Bootloader**: long-lived Python process, started by Recto at launcher
spawn time when any `spec.secrets[].source == "enclave"`. Holds the
public key for each enclave-resident secret. Maintains a ring buffer of
pending sign requests, a session-token cache keyed by `(service,
secret, child-pid)`, and the HTTPS server that the phone polls. Speaks
to the launcher over a local socket.

**Launcher**: existing `recto.launcher`. Detects `SigningCapability`
fetch results and routes them to the bootloader instead of injecting
as env vars. Exposes a local sign-helper socket to the supervised
child; child apps that opt in call `sign(message)` and use the
resulting signature.

**Supervised child**: opt-in only. Apps that want enclave-gated
crypto open the local sign-helper socket the launcher exposes, send a
`SignRequest`, receive a `SignResponse`. Apps that don't want this
keep using env-var-injected secrets (DirectSecret backends).

## Cryptographic primitives

- **Signature schemes** &mdash; per-pairing, negotiated via
  `supported_algorithms` in the registration request. The phone
  declares one algorithm at registration time; the bootloader records
  it alongside the public key and verifies all subsequent signatures
  with the same algorithm. v0.4.0 supports two:
  - **`ed25519`** (RFC 8032). Public keys 32 bytes; signatures 64
    bytes. Both are raw on the wire (no DER), base64url-encoded in
    JSON envelopes. Phone signs the message bytes directly (Ed25519
    handles internal SHA-512 hashing). Default on Android (StrongBox-
    backed via `KeyPairGenerator` with `setIsStrongBoxBacked(true)`)
    and on Windows / Mac dev hosts (software-backed via BouncyCastle).
  - **`ecdsa-p256`** (FIPS 186-4 + RFC 6979). Public keys 64 bytes
    (X || Y, big-endian, raw &mdash; NO 0x04 uncompressed-point
    prefix, NO DER); signatures 64 bytes (R || S, big-endian, raw
    &mdash; NO DER). Phone SHA-256-hashes the message before signing.
    Default on iOS, where the Secure Enclave is available via
    `SecKeyCreateRandomKey` with `kSecAttrTokenIDSecureEnclave` only
    for ECDSA P-256 &mdash; the Secure Enclave does NOT natively
    support Ed25519 as of iOS 18. Apple's `SecKeyCreateSignature`
    returns DER-encoded signatures by default; the phone converts
    DER to raw R || S before sending.
- **Hash for signed payloads**: BLAKE2b-256 of the canonicalized JSON
  payload (sorted keys, no whitespace, UTF-8). Phone signs the hash,
  not the raw payload, so JSON encoding differences across clients
  don't break verification. Note that for the registration challenge
  specifically, the message is already 32 random bytes (not JSON), so
  no canonicalization is needed &mdash; the phone signs the decoded
  challenge bytes directly.
- **Session token**: JWT (RFC 7519) signed with EdDSA (alg=`EdDSA`,
  per RFC 8037). Claims: `iss` = phone-public-key fingerprint,
  `sub` = `{service}:{secret}`, `aud` = bootloader-id, `exp` = unix
  ts, `iat` = unix ts, `jti` = uuid4, `recto:scope` = `["sign"]`,
  `recto:max_uses` = int. The phone signs the JWT once during a
  session-issuance request; the bootloader caches and replays it for
  subsequent sign operations until expiry.
- **No symmetric encryption on the wire**. HTTPS handles transport
  encryption end-to-end. Payloads are signed but not encrypted (the
  plaintext-on-wire is fine because TLS protects it; if TLS fails the
  signature still proves authenticity).

## HTTPS endpoints (bootloader)

The bootloader runs an HTTPS server bound to the host that the phone
can reach. For personal-use deployments, this is typically:

- **LAN-only** (phone on same WiFi as server): `https://<server-lan-ip>:8443`.
- **Cloudflare Tunnel** (phone over the internet): `https://recto-bootloader.<your-domain>` fronted by `cloudflared`, gated by Cloudflare Access. The bootloader trusts the
  CF-Access-Authenticated-User-Email header for auth; only the operator's
  email is allowed.

TLS material: self-signed cert pinned by the phone app on first
registration (LAN case), or operator's real cert via Cloudflare (tunnel
case). The phone app pins the bootloader's public key at registration
time; subsequent connections that present a different key are refused.

### `POST /v0.4/register`

**Phone -> bootloader** (one-time, during initial pairing).

Request body:

```json
{
  "phone_id": "<random uuid4>",
  "device_label": "Pixel 9 / operator",
  "public_key_b64u": "<32-byte Ed25519 public key, base64url-encoded>",
  "supported_algorithms": ["ed25519"],
  "v0_4_protocol": 1,
  "registration_proof": {
    "challenge": "<server-issued challenge from /v0.4/registration_challenge>",
    "signature_b64u": "<phone's Ed25519 signature over the challenge>"
  }
}
```

Response:

```json
{
  "registered": true,
  "phone_id": "<echoed back>",
  "bootloader_id": "<random uuid4 the bootloader assigned>",
  "managed_secrets": [
    {"service": "myservice", "secret": "MY_SIGNING_KEY", "algorithm": "ed25519"}
  ]
}
```

Bootloader stores the phone's public key in `~/.recto/bootloader/phones.json`
(human-readable, ACL-protected to operator-only on Linux/macOS,
DPAPI-machine encrypted on Windows). Subsequent operations are
authenticated by verifying signatures against this stored public key.

### `GET /v0.4/registration_challenge`

**Phone -> bootloader** (one-time, immediately before `/v0.4/register`).

Response:

```json
{
  "challenge_b64u": "<32 random bytes>",
  "expires_at_unix": 1714175400
}
```

The challenge is single-use, expires after 60 seconds. Prevents replay
of registration requests.

### `POST /v0.4/issue_session`

**Phone -> bootloader** (after the operator approves a session-issuance
request via the phone UI).

Request body:

```json
{
  "phone_id": "<from registration>",
  "session_token_jwt": "<EdDSA-signed JWT, see Cryptographic primitives above>"
}
```

The JWT's claims define the session scope (which `{service}:{secret}`
combination) and lifetime (`exp`). Bootloader verifies the signature
against the registered public key, then caches the JWT keyed by
`(service, secret)`.

Response:

```json
{
  "session_id": "<bootloader-assigned uuid>",
  "expires_at_unix": 1714175400
}
```

### `GET /v0.4/pending?phone_id=<phone_id>`

**Phone -> bootloader** (poll after wakeup push, or on app foreground).
The <c>phone_id</c> query parameter identifies which phone is asking;
the bootloader filters its pending-request buffer to the requests
queued for that phone. (For single-phone deployments the parameter is
still required so the same code path works for v0.6+ multi-phone
federation.)

Response:

```json
{
  "requests": [
    {
      "request_id": "<uuid>",
      "kind": "session_issuance" | "single_sign",
      "service": "myservice",
      "secret": "MY_SIGNING_KEY",
      "context": {
        "child_pid": 12345,
        "child_argv0": "python.exe",
        "requested_at_unix": 1714175123,
        "operation_description": "Issue 24h session for myservice/MY_SIGNING_KEY",
        "payload_hash_b64u": "<BLAKE2b-256 of the data to be signed>"
      }
    }
  ]
}
```

Phone displays `operation_description` and the `payload_hash_b64u`
(truncated to 16 hex chars for human comparison) in the approval UI.
The operator visually confirms the hash matches what they expect
before approving.

### `POST /v0.4/respond/<request_id>`

**Phone -> bootloader** (after operator approves or denies).

Approve:

```json
{
  "phone_id": "<from registration>",
  "decision": "approved",
  "signature_b64u": "<64-byte signature over the payload hash, raw form per the phone's algorithm>"
}
```

Deny:

```json
{
  "phone_id": "<from registration>",
  "decision": "denied",
  "reason": "operator declined" | "biometric failed" | "context mismatch"
}
```

The <c>phone_id</c> field tells the bootloader which registered phone
to look up the public key for during signature verification. The
algorithm is the one stored at registration time for that phone (see
"Cryptographic primitives" above &mdash; <c>ed25519</c> raw 64-byte
signatures, or <c>ecdsa-p256</c> raw 64-byte R || S).

Bootloader verifies the signature against the registered public key,
then either resolves the pending sign request (approved) or surfaces
the denial to the launcher's local-socket consumer (denied).

### `POST /v0.4/notify` (internal)

**Bootloader -> push gateway** (internal, not phone-facing).

When a new request lands in the pending buffer, the bootloader sends
a wake-up push notification:

- **iOS**: APNs alert with `content-available: 1` and a tiny payload
  `{"recto.kind": "wake"}`. App handler triggers a `/v0.4/pending` poll.
- **Android**: FCM data message with the same payload. App handler
  same as iOS.

The push payload contains NO sensitive data; the actual sign-request
context is fetched only over HTTPS via `/v0.4/pending`.

Once a session JWT is cached:

```
Child process              Launcher           Bootloader            Phone
     │                        │                   │                  │
     │  sign("hello")         │                   │                  │
     ├───────────────────────►│                   │                  │
     │                        │ sign(payload, JWT)│                  │

     │                        │                   │ sign locally     │

     │  signature             │                   │                  │
     │◄───────────────────────┤                   │                  │
```

In session-cached operation, the phone is NOT involved (latency stays
low; phone battery is preserved). The session JWT's `exp` and
`max_uses` claims define when the next phone-mediated re-issuance is
required.

## Session-token model

The session JWT is the load-bearing optimization. Without it, every
sign would require a phone wakeup + biometric (~10-20s latency, awful
UX). With it, only the FIRST sign in a session requires phone
interaction; subsequent signs go through the bootloader's cached JWT
verification.

- **JWT signing key**: the phone's Ed25519 enclave key. Same key signs
  the JWT and the per-operation responses.
- **JWT verification key**: the bootloader has the public key; verifies
  the JWT against it on every sign request.
- **Default lifetime**: 24 hours. Configurable per-secret via
  `service.yaml`'s `spec.secrets[].config.session_lifetime_seconds`.
  Maximum: 7 days. Minimum: 60 seconds (effectively no caching).
- **Default max_uses**: 1000 sign operations per session. Configurable
  via `config.session_max_uses`. Set to 0 (unlimited) only for
  development.
- **Renewal**: when 80% of `exp` or 80% of `max_uses` is consumed, the
  bootloader proactively requests a new session JWT from the phone.
  This avoids latency spikes at renewal time (the new JWT arrives
  before the old one is exhausted).
- **Revocation**: the operator can deny pending sessions from the phone
  app. Bootloader honors the denial immediately; in-flight signs that
  were using the old session continue with the old session until that
  session's natural expiry (no mid-flight invalidation).

## Key onboarding flow (one-time per phone)

1. **Operator triggers registration on the bootloader**: `recto v0.4
   register` CLI subcommand. Bootloader generates a one-time pairing
   code (6 digits, 5-minute TTL) and prints it.
2. **Operator opens the phone app**: app shows "Pair with bootloader"
   screen, prompts for the pairing code.
3. **Operator enters the code on the phone**: phone app reaches out to
   `https://<bootloader>/v0.4/registration_challenge?code=<code>`,
   bootloader validates the code and returns a fresh challenge.
4. **Phone generates Ed25519 keypair in enclave**: `Microsoft.Maui.Authentication.SecureStorage`
   wraps the platform-specific generation
   (iOS `SecKeyCreateRandomKey` with `kSecAttrTokenIDSecureEnclave`;
   Android `KeyGenerator` with `setUserAuthenticationRequired(true)`
   and `setIsStrongBoxBacked(true)`). Public key extracted; private
   key stays in enclave forever.
5. **Phone signs the challenge with the new key**: biometric prompt
   gates the sign. Operator approves.
6. **Phone POSTs `/v0.4/register`**: with the public key + signed
   challenge. Bootloader verifies, stores the public key, returns the
   list of `managed_secrets` the operator has configured for this
   phone.
7. **Phone displays "Paired"**: shows the bootloader's `bootloader_id`
   and the list of managed secrets. Operator can now approve sign
   requests.

After pairing, the phone+bootloader pair is bound. The bootloader will
not accept registration from a different phone for the same managed
secrets without operator-side rotation.

## Failure modes

| Failure | Detection | Recovery |
|---|---|---|
| Phone offline at session-issuance time | HTTP timeout on `/v0.4/issue_session` | Fail the sign with `EnclaveUnavailableError`; child decides whether to retry or fail. No fallback to env-var. |
| JWT expired mid-operation | Bootloader checks `exp` before every sign | Bootloader transparently re-issues (sees expiry, calls `/v0.4/issue_session`, retries the sign). Latency spike but no operator intervention. |
| Phone refuses biometric | Phone app reports `decision: denied, reason: biometric failed` | Bootloader propagates denial to the launcher; launcher propagates to child as a sign error. Child must surface the failure. |
| Bootloader crashes | Launcher heartbeat to bootloader fails | Launcher restarts the bootloader. Cached sessions are LOST (bootloader state is in-memory). First sign after restart re-prompts the phone. |
| Push notification dropped | Bootloader-side TTL on pending request | Phone app on foreground polls `/v0.4/pending` regardless of push. Push is best-effort wakeup, not delivery guarantee. |
| Operator loses phone | Both phones are pre-registered; lost phone's key is revoked from bootloader | Operator runs `recto v0.4 revoke <phone_id>` on the bootloader. Sign requests now route only to the surviving phone(s). |

## Security model

**Trust boundaries:**

- **Phone enclave** is the trust root. The private key is non-exportable;
  if the phone is stolen, the attacker needs the operator's biometric
  to unlock it.
- **Bootloader** is semi-trusted. It holds public keys, session JWTs,
  and pending requests. An attacker with code-exec on the bootloader
  can: (a) read all in-flight sign requests + cached JWTs; (b) replay
  cached JWTs until their `exp`; (c) NOT extract private keys (those
  are on the phone). Mitigation: short JWT lifetimes (default 24h);
  the operator can manually revoke any session from the phone app.
- **Phone-bootloader transport (HTTPS)** is encrypted by TLS. Public
  key pinning prevents MITM by a CA compromise.
- **Push notifications** are NOT trusted. Payload contains no sensitive
  data; the phone re-fetches actual request context over HTTPS.

**Threats explicitly NOT mitigated by v0.4.0:**

- **Compromised phone OS** can read enclave-released signatures
  in-flight (after the enclave hands them to the userspace OS for
  delivery to the phone app). Mitigation depends on platform OS
  security; Recto can't compensate.
- **Bootloader running as the wrong user account** could expose cached
  JWTs to other processes on the same machine. Mitigation: bootloader
  must run under a dedicated service account; never root/admin/SYSTEM.
- **Operator approves a malicious sign request without reading the
  context.** The phone app shows `operation_description` and the
  truncated `payload_hash_b64u` for visual confirmation; the operator
  is the last line of defense against bootloader compromise.

## Implementation guidance

**Phone app (MAUI Blazor):**

- Biometric prompts are platform-native:
  - iOS: `LAContext` (LocalAuthentication framework). Trigger via
    `SecAccessControl` flags `.biometryCurrentSet | .privateKeyUsage`
    bound to the `SecKey` at generate time; every signing operation
    auto-prompts. Manifest: `NSFaceIDUsageDescription` in `Info.plist`.
  - Android: `BiometricPrompt`. Trigger via `KeyGenParameterSpec`
    `setUserAuthenticationRequired(true)` +
    `setUserAuthenticationParameters(timeoutSec, type)` bound to the
    keystore alias at generate time. Manifest:
    `android.permission.USE_BIOMETRIC` in `AndroidManifest.xml`.
- iOS keypair: `SecKey` APIs for ECDSA P-256 with
  `kSecAttrTokenIDSecureEnclave`. iOS 14+ for Secure Enclave P-256
  generally; iOS 15+ floor in this app. The Secure Enclave does NOT
  natively support Ed25519 as of iOS 18 &mdash; the algorithm-
  negotiation seam in `supported_algorithms` lets iOS advertise
  `ecdsa-p256` instead. `SecKeyCreateSignature` returns DER-encoded
  signatures; convert to the protocol's raw R||S form (32+32 bytes)
  before sending &mdash; `System.Formats.Asn1.AsnReader` handles the
  DER parse cleanly in &lt; 25 lines.
- Android keypair: `KeyPairGenerator.getInstance("EC",
  "AndroidKeyStore")` with `KeyGenParameterSpec.Builder`,
  `setAlgorithmParameterSpec(new ECGenParameterSpec("secp256r1"))`,
  `setDigests(KeyProperties.DIGEST_SHA256)`,
  `setIsStrongBoxBacked(true)`, `setUserAuthenticationRequired(true)`.
  Sign with `Signature.getInstance("SHA256withECDSA")` &mdash; returns
  DER, convert to raw R||S the same way as iOS. AndroidKeyStore does
  NOT support Ed25519 natively (no
  `KeyProperties.KEY_ALGORITHM_ED25519` constant) so Android phones
  also advertise `ecdsa-p256` like iOS. On devices without StrongBox
  hardware, catch `StrongBoxUnavailableException` and retry without
  the StrongBox flag &mdash; falls through to TEE-backed (still
  hardware-isolated, just not the dedicated HSM).
- HTTP client: `HttpClient` with custom `HttpClientHandler` for the
  bootloader's pinned cert.
- Push: `Plugin.LocalNotification` (iOS) / `Firebase.Messaging`
  (Android). Tokens registered with the bootloader at pairing time.

**Bootloader (Python, lives in `recto.bootloader`):**

- HTTPS server: stdlib `http.server` + `ssl.SSLContext`, no extra deps.
- JWT: `pyjwt` (optional dep, lazy-loaded). Verify with `algorithm="EdDSA"`,
  `cryptography` package backing.
- Push notifications: `aioapns` (iOS) and `pyfcm` or stdlib `urllib`
  POST to FCM HTTP v1 (Android). Both as optional deps.
- Persistence: `~/.recto/bootloader/{phones,sessions,pending}.json`,
  ACL-tightened to operator user.
- Cross-platform: ships in the same wheel as `recto-core`. Optional
  dependencies via `pip install recto[v0.4]`.

**Launcher (existing `recto.launcher`):**

- Detects `SigningCapability` returns from `SecretSource.fetch()` (the
  ABC seam already exists, see `recto.secrets.base`).
- Spawns the bootloader subprocess at first SigningCapability fetch,
  parents under launcher (dies with launcher per Job Object / process
  group).
- Exposes a Unix socket (Linux/macOS) or named pipe (Windows) for
  child-process sign requests. Path:
  `/tmp/recto-{service}-sign.sock` or `\\.\pipe\recto-{service}-sign`.
- Forwards `sign(payload)` calls from the child to the bootloader's
  internal RPC interface.

**Child process (consumer):**

- Opt-in only. Apps that want enclave-gated crypto check for
  `RECTO_SIGN_HELPER` env var (set by launcher) and connect to that
  socket/pipe. Apps that don't continue using env-var-injected secrets
  unchanged.
- Reference client: `recto.signhelper` Python module ships with Recto
  and exposes a `sign(message: bytes) -> bytes` callable. Other
  languages (C#, Go, Rust) get protocol docs in
  `docs/v0.4-signhelper-clients.md` (TBD).

## Phone management (v0.5+)

Round-7 expansion. Three management endpoints let the operator drive
multi-phone registration and lost-phone recovery from the surviving
phone's UI rather than requiring shell access to the bootloader.

### Phone-id deduplication

When a previously-registered phone re-pairs (same `phone_id`, possibly
with a fresh public key after enclave wipe / re-enrollment), the
bootloader replaces the existing entry in-place rather than appending
a duplicate. The operator just authorized the re-pair via biometric on
the phone's enclave; the new public key supersedes the old. Phones are
keyed by `phone_id`; uniqueness is the bootloader's invariant.

### `GET /v0.4/manage/phones?phone_id=<self>`

**Phone -> bootloader**. Returns the list of phones registered with
this bootloader OTHER than the requester (the requester already knows
about itself). Used by the surviving phone's UI to render its
"Registered phones" section.

Response:

```json
{
  "phones": [
    {
      "phone_id": "<uuid>",
      "device_label": "Pixel 9 / operator",
      "algorithm": "ed25519",
      "paired_at": "2026-04-26T22:00:00+00:00"
    }
  ]
}
```

### `GET /v0.4/manage/revoke_challenge?phone_id=<self>`

**Phone -> bootloader**. Mints a single-use, 60-second-TTL 32-byte
challenge that the calling phone must sign as the authorization for a
subsequent `POST /v0.4/manage/revoke`. Same shape as the pairing
challenge. Isolating the endpoint by purpose lets the bootloader
reject reuse: a pairing challenge cannot be replayed to authorize a
revocation.

Response:

```json
{
  "challenge_b64u": "<32 random bytes, base64url>",
  "expires_at_unix": 1714175400
}
```

### `POST /v0.4/manage/revoke`

**Phone -> bootloader**. Revokes a target phone's registration. The
revoking phone signs the challenge bytes (decoded from base64url) with
its enclave key; the bootloader verifies the signature against the
revoking phone's stored public key + algorithm. On success, the target
phone is removed from the registered-phones roster, any pending
requests targeting it are dropped, and any active revoke-challenge for
it is cleared. In-flight requests already-fetched-but-not-responded
fail when the revoked phone tries to respond (404 unknown request_id).

Request body:

```json
{
  "revoking_phone_id": "<self>",
  "target_phone_id": "<phone to revoke>",
  "challenge": "<echoed from /v0.4/manage/revoke_challenge>",
  "signature_b64u": "<phone's signature over the decoded challenge bytes>"
}
```

Response:

```json
{
  "revoked": true,
  "target_phone_id": "<echoed>"
}
```

In v0.4.0 a phone can revoke any other phone registered with the same
bootloader (single-operator assumption from the original RFC). v0.6+
multi-user models tighten authorization rules (the revoking phone
might need to be on an admin allowlist for the operator account).

A phone cannot revoke itself; the bootloader returns 400. To unpair
yourself, use the `Unpair` button on the phone (which clears local
state without calling the bootloader; the registration entry remains
for the next phone to revoke).

## Capability JWTs (v0.5+)

Round-6 expansion. The phone's enclave key now signs JWT capabilities
that authorize the bootloader (or an external agent) to perform sign
operations within a scoped, time-bounded, revocable window. This is
the primitive that unlocks both the bootloader-internal latency
optimization (cache the JWT, skip phone wakeup for repeat signs within
the validity window) and the agent path (issue a JWT to an external
bearer; the bearer presents it; the bootloader verifies). The same
artifact serves both bearers.

Wire-protocol delta is purely additive &mdash; the
`PendingRequest.kind` field gains a `session_issuance` value, and
`PendingRequestContext` gains optional fields populated only for that
kind. Protocol stays at version 1.

### `session_issuance` request kind

Bootloader -> phone, via the existing `/v0.4/pending` queue. Asks the
operator to approve a JWT capability for `service`/`secret` with the
specified scope, lifetime, and use limit.

`PendingRequestContext` populates:

- `session_bearer` (string) &mdash; <c>"bootloader"</c> for
  bootloader-cached use, or <c>"agent:&lt;agent-id&gt;"</c> for an
  external agent
- `session_scope` (string array) &mdash; e.g. <c>["sign"]</c>
- `session_lifetime_seconds` (int) &mdash; default 24h, max 7d, min 60s
- `session_max_uses` (int) &mdash; default 1000, set to 0 only in dev
- `session_bootloader_id` (string) &mdash; the JWT's <c>aud</c> claim;
  prevents JWTs from one bootloader being replayed at a different one

### JWT format

Standard JWS compact form: `base64url(header).base64url(claims).base64url(signature)`.

Header:

```json
{ "alg": "EdDSA" | "ES256", "typ": "JWT" }
```

`alg` is `EdDSA` for Ed25519-keyed phones (Android StrongBox,
Windows-software dev), `ES256` for ECDSA P-256-keyed phones (iOS
Secure Enclave). The phone picks based on its enclave's algorithm
(advertised at registration time as `supported_algorithms[0]`).

Claims:

```json
{
  "iss": "<base64url SHA-256 of phone public key>",
  "sub": "<service>:<secret>",
  "aud": "<bootloader_id>",
  "exp": <unix ts>,
  "iat": <unix ts>,
  "jti": "<uuid4>",
  "recto:scope": ["sign"],
  "recto:max_uses": 1000,
  "recto:bearer": "bootloader" | "agent:<agent-id>"
}
```

Signature is the phone's enclave-key signature over
`base64url(header) + "." + base64url(claims)` &mdash; the same wire
shape as Ed25519 / ECDSA P-256 raw R||S that `single_sign` uses.

### Approval response

The phone approves the request and POSTs the signed JWT back via
`/v0.4/respond/{request_id}` with:

```json
{
  "phone_id": "<from registration>",
  "decision": "approved",
  "session_jwt": "<base64url-joined JWT>"
}
```

The bootloader verifies the JWT's signature against the phone's
registered public key, checks `aud` matches its own
`bootloader_id`, validates `exp > now`, and stores the JWT keyed by
`{service, secret, recto:bearer}`. For bearer = `bootloader`,
subsequent sign operations within the validity window replay the JWT
locally (no phone wakeup). For bearer = `agent:...`, the bootloader
hands the JWT to the agent through whatever channel the agent uses to
fetch its capabilities (out of scope for this RFC; v0.6+ defines that
channel).

### Why session_issuance subsumes the originally-planned `/v0.4/issue_session`

The earlier RFC sketch had a separate `/v0.4/issue_session` endpoint.
That's been folded into the existing `/v0.4/pending` -> `/v0.4/respond`
flow with `kind = "session_issuance"` because the operator-approval
and biometric-gating semantics are identical to other pending
requests. One protocol shape covers all four kinds (single_sign,
totp_provision, totp_generate, session_issuance), reducing the
phone-side and bootloader-side code paths.

## TLS pinning (v0.5+)

Round-6 hardening. After successful pairing, the phone records the
SPKI hash (SHA-256 of the bootloader's `SubjectPublicKeyInfo`,
base64url-encoded with no padding) of the bootloader's TLS cert in
its persistent pairing record (`PairingState.BootloaderSpkiPin`).
Every subsequent connection verifies the bootloader's cert SPKI
against this pin before completing the TLS handshake; mismatch fails
the connection regardless of the system trust store's outcome.

This is the standard SSH "trust-on-first-use" model:

- **First pair**: the phone's `HttpClientHandler.ServerCertificateCustomValidationCallback`
  consults the system trust store (Cloudflare Tunnel-fronted bootloaders
  have valid cert chains; self-signed LAN deployments fail at this
  step in v0.5 &mdash; future round can add a "trust this self-signed
  cert" prompt at pairing time). On success the observed SPKI is
  recorded and promoted to a verification pin.
- **Subsequent connections**: the same handler compares the
  presented cert's SPKI against the stored pin. Pin match: accept.
  Pin mismatch: refuse, regardless of system trust. This blocks
  CA-compromise MITM attacks because the attacker would have to
  obtain a cert with the exact SPKI of the original, which requires
  the original private key.

Pin format: `<base64url(SHA-256(SubjectPublicKeyInfo))>` &mdash; same
form HPKP and Chrome's pinset use, so existing tooling (`openssl x509
-pubkey -noout | openssl asn1parse | sha256sum`) produces compatible
output for inspection.

Pin invalidation: changing the bootloader's cert (renewal, key
rotation, etc.) breaks all paired phones. The recovery path is for
the operator to unpair + re-pair each phone &mdash; same as if a new
bootloader were stood up. v0.7+ may add a "pre-publish" mechanism
where the bootloader announces an upcoming SPKI and phones accept
either old or new during a rollover window.

## TOTP capability (v0.5)

Round-5 expansion. The phone-resident vault gains a second credential
type alongside the v0.4 signing keypair: TOTP shared secrets per
RFC 6238. Wire-protocol delta is purely additive &mdash; the
`PendingRequest.kind` field gains two new values, and
`PendingRequestContext` gains optional fields populated only when the
relevant kind is set.

### `totp_provision` request kind

Bootloader -> phone, via the existing `/v0.4/pending` queue. Imports a
shared TOTP secret to the phone's local SecureStorage. Operator
approves with biometric / device passcode (the standard pending-request
approval flow); on approval the phone stores the secret keyed by
`totp_alias` and returns success.

`PendingRequestContext` populates:

- `totp_alias` (string) &mdash; the storage key (e.g. `"myservice.totp.aws_console"`)
- `totp_secret_b32` (string) &mdash; RFC 4648 base32-encoded shared secret (padding stripped)
- `totp_period_seconds` (int, optional, default 30)
- `totp_digits` (int, optional, default 6)
- `totp_algorithm` (string, optional, default `"SHA1"`; also `"SHA256"` / `"SHA512"`)

Approval response: `{ "phone_id": "...", "decision": "approved" }`
&mdash; no extra fields. Denial: `{ "phone_id": "...", "decision": "denied",
"reason": "..." }`.

### `totp_generate` request kind

Bootloader -> phone. Asks the phone to compute the current TOTP code
for a previously-provisioned alias.

`PendingRequestContext` populates:

- `totp_alias` (string) &mdash; must match a `totp_provision` previously approved by this phone

Approval response: `{ "phone_id": "...", "decision": "approved",
"totp_code": "012345" }` &mdash; the current code, zero-padded to the
configured digit count. Denial: standard.

The phone computes the code locally from its stored secret + current
UTC time per RFC 6238. The bootloader then forwards the code to
whatever needs it (typically a supervised child process or an
authorized agent), or in the round-5 mock, verifies it against its
server-side mirror of the secret (with a +-1 time-step window for
clock skew).

### Why TOTP belongs in the universal vault

TOTP shared secrets are the most-deployed second factor in the world.
Today they live in one of:

- Phone-only authenticator apps (Authy, Google Authenticator) &mdash;
  locked to one device, painful to migrate.
- Cloud-synced authenticator apps (Authy cloud, 1Password) &mdash; the
  cloud vendor sees plaintext shared secrets at sync time.
- Password managers with a TOTP feature (1Password, Bitwarden) &mdash;
  same as cloud-synced.

Recto's pitch for this slot: the phone's SecureStorage holds the
secret; the bootloader is a transient relay during provisioning (the
operator scans a QR on a desktop tool, the bootloader forwards the
secret to the phone for permanent storage); the bootloader forgets the
secret after provisioning. No cloud sync provider sees plaintext.
Cross-phone migration in v0.6+ uses operator-approved re-provisioning
between phones rather than third-party sync.

## Ethereum signing capability (v0.5+)

Recto's universal vault gains an `eth_sign` credential kind. The
`PendingRequest.kind` field accepts a new value, and
`PendingRequestContext` carries Ethereum-specific fields populated when
the relevant kind is set. The phone holds a BIP39 mnemonic in platform
secure storage; per-request, it derives the secp256k1 private key at
the requested BIP32 path, computes the EIP-191 / EIP-712 / RLP digest,
signs, and returns a 65-byte r||s||v signature. The Python launcher
tier never holds the private key — only the phone derives, signs, and
discards.

### `eth_sign` request kind

`PendingRequestContext` populates:

- `eth_chain_id` — integer chain id (1 = mainnet, 8453 = Base,
  11155111 = Sepolia, etc.). Operator UI surfaces this so the operator
  can refuse a mainnet signing request that should have been on a
  testnet.
- `eth_message_kind` — discriminator: `"personal_sign"` (EIP-191
  prefixed message), `"typed_data"` (EIP-712 structured data), or
  `"transaction"` (EIP-1559 / 2930 / legacy RLP-encoded tx; reserved).
- `eth_address` — expected lowercase 0x-prefixed signer address.
  The phone refuses the request if the BIP32 derivation produces a
  different address.
- `eth_derivation_path` — BIP32/BIP44 path the phone should derive
  the signing key at (default `"m/44'/60'/0'/0/0"`). Allows
  multi-account setups (`m/44'/60'/0'/0/N` for the N-th account in
  the same mnemonic).
- Exactly one of:
  - `eth_message_text` (for `personal_sign`) — the human-readable
    string the operator approved seeing on screen.
  - `eth_typed_data_json` (for `typed_data`) — the EIP-712 JSON spec
    (domain + types + primaryType + message).
  - `eth_transaction_json` (for `transaction`) — RLP fields. Reserved
    for a follow-up; not produced by the bootloader yet.

### Approval response

`RespondRequest` populates:

- `eth_signature_rsv` — 65-byte r||s||v secp256k1 signature as a hex
  string with `0x` prefix. Canonical low-s form per Ethereum's
  signature acceptance rules.
- `signature_b64u` — also populated, containing the phone's
  registration-key signature over the response body. The bootloader
  verifies this in `_handle_respond` so the response is provably from
  the paired phone (same guarantee `single_sign` provides). The
  Ethereum-side `eth_signature_rsv` is forwarded to the consumer
  (smart contract / off-chain verifier) as opaque bytes; the bootloader
  does not validate the Ethereum signature itself.

### Why ETH belongs in the universal vault

Hot wallets (browser-extension MetaMask, mobile MetaMask, Coinbase
Wallet) are vulnerable to:

- Browser extension supply-chain compromises (a malicious extension
  update reads the unlocked vault).
- Phishing pages requesting `eth_sign` / `personal_sign` with attacker-
  crafted message text designed to look like an innocuous login.
- Clipboard hijackers replacing copied recipient addresses.
- Malware on the same machine reading the wallet password from key
  loggers and decrypting the local vault file.

Cold wallets (Ledger / Trezor) solve the storage problem but introduce
device-management overhead and are awkward for high-frequency
operations (every signature requires reaching for the device).

Recto's pitch for this slot: the phone holds the mnemonic in
SecureStorage; the bootloader is the transient relay; each signing
operation requires biometric approval on the phone with the message
displayed in plaintext (`personal_sign`) or rendered structurally
(`typed_data`); agent-driven signing flows through capability JWTs
whose scope claims encode a per-operation cap (target contract, method
selector, value cap, gas cap, expiry) — operator approves the
capability once, the agent draws against it within its bounds, the
capability auto-expires.

The Python launcher tier ships verify-side helpers in
`recto.ethereum` (Keccak-256, EIP-191 hashing, secp256k1 ECDSA verify
+ public-key recovery, EIP-55 checksum addresses) — pure stdlib, no
new packages required, gated behind the `recto[ethereum]` extra. Sign
side is phone-only; private keys never reach the launcher.

## Bitcoin signing capability (v0.5+)

Recto's universal vault gains a `btc_sign` credential kind alongside
`eth_sign`. The two cryptocurrency credential kinds share more than
they differ: same BIP-39 mnemonic (one phone-resident master secret
recoverable in any BIP-39 wallet), same secp256k1 curve, same
operator-approval ceremony, same JWT-capability delegation pattern
for agent flows. The differences are the BIP-44 path tree
(`m/84'/0'/0'/0/N` for native-SegWit P2WPKH default vs Ethereum's
`m/44'/60'/0'/0/N`), the address-encoding scheme (bech32 / bech32m
vs Ethereum's last-20-bytes-of-keccak), and the digest preimage
(BIP-137 prefixed-and-varint-encoded message vs EIP-191 prefix).

### `btc_sign` request kind

`PendingRequestContext` populates:

- `btc_network` — string discriminator: `"mainnet"`, `"testnet"`,
  `"signet"`, or `"regtest"`. Determines the bech32 HRP / Base58Check
  version byte the phone uses when displaying the expected signing
  address to the operator.
- `btc_message_kind` — discriminator: `"message_signing"` (BIP-137
  compact-signature signed-message verb) or `"psbt"` (BIP-174
  partially-signed Bitcoin transaction; reserved for a follow-up).
- `btc_address` — expected signer address. Bech32 lowercase for
  P2WPKH (`bc1q...` / `tb1q...`), Base58Check for legacy P2PKH
  (`1...` / `m...` / `n...`) or nested SegWit P2SH-P2WPKH
  (`3...` / `2...`). The phone refuses the request if the BIP32
  derivation produces a different address.
- `btc_derivation_path` — BIP32/BIP44 path the phone resolves the
  signing key from. Default `"m/84'/0'/0'/0/0"` (native-SegWit
  account 0, change 0, index 0). Allows multi-account setups
  (`m/84'/0'/0'/0/N`) and alternate address kinds via different
  purpose levels (`m/49'/0'/0'/0/N` for nested SegWit,
  `m/44'/0'/0'/0/N` for legacy P2PKH).
- Exactly one of:
  - `btc_message_text` (for `message_signing`) — the human-readable
    string the operator approved seeing on screen.
  - `btc_psbt_base64` (for `psbt`) — base64-encoded BIP-174 PSBT.
    Reserved for a follow-up; not produced by the bootloader yet.

### Approval response

`RespondRequest` populates:

- `btc_signature_base64` — 65-byte BIP-137 compact signature
  (`header_byte || r || s`) base64-encoded (88 chars typical). The
  header byte encodes recovery id + intended address kind per
  BIP-137 §"Header byte values":
  - `27..30` → P2PKH uncompressed (recovery id = header − 27)
  - `31..34` → P2PKH compressed   (recovery id = header − 31)
  - `35..38` → P2SH-P2WPKH        (recovery id = header − 35)
  - `39..42` → P2WPKH native-SegWit (recovery id = header − 39)
- `signature_b64u` — also populated, containing the phone's
  registration-key Ed25519 signature over the response body. The
  bootloader verifies this in `_handle_respond` so the response is
  provably from the paired phone (same guarantee `eth_sign`
  provides). The Bitcoin-side `btc_signature_base64` is forwarded to
  the consumer (smart contract / off-chain verifier / wallet
  performing on-chain verification) as opaque bytes; the bootloader
  does not validate the secp256k1 signature itself.

### Why BTC belongs in the universal vault

The same threat-model arguments apply as for ETH:

- Hot wallets (browser-extension, mobile-software wallets) are
  vulnerable to extension supply-chain compromises, phishing pages
  spoofing legitimate signing prompts, clipboard hijackers, and
  malware reading the local vault password.
- Cold wallets (Ledger / Trezor) solve storage but introduce
  device-management overhead and don't pair cleanly with mobile
  workflows.

Recto's pitch: phone holds the mnemonic in `SecureStorage`, every
signing operation requires biometric approval with the message
displayed in plaintext (`message_signing`) or rendered structurally
(`psbt` — future), and agent-driven flows go through capability
JWTs whose scope claims encode a per-operation cap (target output,
value cap, fee cap, expiry).

The Python launcher tier ships verify-side helpers in `recto.bitcoin`
(RIPEMD-160 + HASH160, double-SHA-256, bech32 / bech32m encoding,
BIP-137 signed-message hashing, P2WPKH / P2PKH / P2SH-P2WPKH address
derivation, BIP-137 compact-signature parse + recover) — pure stdlib,
no new packages, gated behind the `recto[bitcoin]` extra. The
secp256k1 verify primitive is delegated to `recto.ethereum` (same
curve). Sign side is phone-only; private keys never reach the
launcher.

## Versioning and forward-compat

This RFC is version `1` of the v0.4 protocol. The `v0_4_protocol`
field in the registration request lets phone and bootloader negotiate.
Breaking changes to the wire format bump this number; minor additive
changes (new optional fields) keep version `1`.

Future versions add:

- **v0.5**: post-quantum signature algorithms (Dilithium / Falcon /
  SPHINCS+). The `algorithm` field in registration's
  `supported_algorithms` enumerates what the phone can do; the
  bootloader picks the highest mutually-supported.
- **v0.6**: m-of-n co-signers. The bootloader holds N phone
  registrations; operations require M approvals.
- **v0.7**: cold-storage backup keys. Operator-driven onboarding
  ceremony with a hardware wallet (Trezor / Ledger) signs a
  recovery package.
